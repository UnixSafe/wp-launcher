# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A single-module Android launcher app that recreates the Windows Phone / Metro "Live Tiles" home screen experience (Kotlin + Jetpack Compose). It replaces the device home screen, lists/launches installed apps, and includes a "Cortana" chat assistant backed by the Gemini API. All user-facing strings are in **French** — keep new UI text French to match.

Note: `namespace`/package is `com.example`, but the published `applicationId` is `com.aistudio.wplauncher.zypkbx`.

## Build & test commands

There is **no Gradle wrapper committed** — use an installed `gradle` or Android Studio's Gradle integration (`./gradlew` only if a wrapper is added later).

- `gradle :app:assembleDebug` — build the debug APK
- `gradle :app:testDebugUnitTest` — run all JVM tests (JUnit + Robolectric + Roborazzi screenshots)
- `gradle :app:testDebugUnitTest --tests "com.example.GreetingScreenshotTest"` — run a single test class
- `gradle :app:connectedDebugAndroidTest` — run instrumentation tests on a connected device/emulator
- `gradle :app:assembleRelease` — release build; signing reads `KEYSTORE_PATH` / `STORE_PASSWORD` / `KEY_PASSWORD` from the environment (key alias `upload`)

Roborazzi screenshot tests compare against PNGs in `app/src/test/screenshots/`. Record/update with `gradle :app:recordRoborazziDebug` and verify with `gradle :app:verifyRoborazziDebug`.

### Gemini API key

The Cortana feature needs a key. The Secrets Gradle Plugin reads `GEMINI_API_KEY` from a `.env` file (copy `.env.example`) and exposes it as `BuildConfig.GEMINI_API_KEY`. When the key is missing or still the placeholder `MY_GEMINI_API_KEY`, `LauncherViewModel.sendCortanaPrompt` falls back to canned local responses instead of calling the network — so the app builds and runs without a key.

## Architecture

**Single-Activity, single-ViewModel MVVM with no navigation library.**

- `MainActivity` is the only Activity. It builds the dependency graph by hand in `onCreate` (`AppDatabase.getDatabase` → `LauncherRepository` → `ViewModelFactory` → `LauncherViewModel`), then hosts the whole UI in one `setContent`. There is **no Navigation Compose** — screen switching is a `when (activeScreen)` block over the `ActiveScreen` enum (`START`, `ALL_APPS`, `SETTINGS`, `CORTANA`). MainActivity also layers overlays on top of the active screen: status bar, Action/Notification Center drawer, toast banner, lock screen, app-open animation, and the default-launcher dialog.

- `LauncherViewModel` (`viewmodel/`) is an `AndroidViewModel` that owns **all** UI state as `StateFlow`/`SharedFlow`. Screens receive the `viewModel` directly and read state via `collectAsState()` — there is no per-screen state holder. When adding behavior, add a `StateFlow` + mutator method here rather than putting business logic in a composable. Manual DI: there is no Hilt/Koin; everything is wired through `ViewModelFactory`.

- `LauncherRepository` (`repository/`) wraps `LauncherDao` for tile/settings persistence **and** queries `PackageManager` for installed apps via `getInstalledApps()` (returns `AppItem` with a rasterized icon `Bitmap`). This is the only place that touches the OS package list; it requires the `QUERY_ALL_PACKAGES` permission declared in the manifest.

- `data/` is the Room layer: `AppDatabase` (db name `launcher_db`) with two entities — `TileEntity` (home tiles, autogenerated id) and `SettingsEntity` (a **singleton** row pinned to `id = 1`). The DB uses `fallbackToDestructiveMigration()`, so **any entity schema change wipes user data** unless you write a real migration; bump the `@Database(version = ...)` when you change a schema.

- `data/GeminiService.kt` is a standalone Retrofit/Moshi/OkHttp client (`GeminiClient.apiService`) that POSTs directly to `generativelanguage.googleapis.com` (model `gemini-3.5-flash`) with the API key as a query param. Note: `firebase-ai` is also a declared dependency, but the **active** Cortana path is this hand-rolled Retrofit client, not Firebase.

- `ui/screens/` holds the Compose screens (`StartScreen`, `AllAppsScreen`, `SettingsScreen`, `CortanaScreen`) and overlay composables (`ActionCenter`, `LockScreen`, `AppOpenOverlay`, `MetroDefaultLauncherDialog`). `ui/theme/` holds Metro color/typography.

### Tiles model

Home tiles are either **built-in pseudo-apps** whose `packageName` starts with `wp:` (e.g. `wp:phone`, `wp:cortana`, `wp:settings`, `wp:weather`) or **real installed apps** (real package + class name). `LauncherViewModel.launchTile` branches on the `wp:` prefix: built-ins fire system intents or switch screens; real tiles build a launcher `Intent` from package/class. Default tiles are seeded once in `checkAndProvisionDefaultTiles()` only when the tile table is empty. Tile sizes are the strings `SMALL` / `MEDIUM` / `WIDE`. A "live tiles" coroutine heartbeat in the ViewModel periodically mutates mock weather/calendar tile text to fake dynamic content.

### Launcher-specific integration

The app registers as a HOME launcher (manifest intent-filter with `CATEGORY_HOME`). On startup it checks `isMyLauncherDefault()` and, if not default, prompts via `MetroDefaultLauncherDialog` → `RoleManager.ROLE_HOME` (with fallbacks for older SDKs). Cortana voice replies use Android `TextToSpeech` (French locale): the ViewModel emits text on `speakEvent` (a `SharedFlow`) and MainActivity collects it to speak.

## Conventions

- Match existing formatting: 4-space indent in Kotlin sources, 2-space indent in Gradle KTS scripts. PascalCase composables/classes; suffixes `Screen`, `ViewModel`, `Repository`, `Entity`, `Dao`.
- All dependency versions live in `gradle/libs.versions.toml` (version catalog) — add/upgrade deps there, reference via `libs.*`. Room and Moshi codegen run through KSP.
- `AGENTS.md` holds the contributor-facing conventions (commit style `type: summary`, PR expectations); follow it for commits and PRs.
